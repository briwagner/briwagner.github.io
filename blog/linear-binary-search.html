Linear and Binary Search

November 21, 2015

Two roads diverged in a wood, and I -
I took the one less traveled by,
And that has made all the difference.

- Robert Frost

Like paths in the woods, we have a choice to make when it comes time to search. Yes, Google or Bing. But also what kind of search we perform. Of course, I didn't know this choice existed until we were asked to select our own Ruby challenge at DBC this week, but that was the choice that has made all the difference for me.

The first kind of search we call linear search. As the traveler proceeds from start to finish, so our programmer hero moves through a collection from first element to last to perform his search. The goal is to test every element for some condition, and when it succeeds, we will kick out of the loop and save unnecessary computer cycles. In Ruby, this can be done with Enumerables. We call these simple methods on an array or hash to run through the entities and perform all kinds of actions. Super clever.

forest = ['path_one', 'path_two', 'path_three', 'path_four']

forest.each { |path| print "Should I take #{path}?" }

Of course, Javascript has its own version of this function, looping through a collection from start to finish, thanks to the for (items in collection) syntax.

When our collections are small, linear search is certainly reasonable to find what we need to find. But think of all the trees in the forest. That's a lot of searching. There must be another way.

Enter binary search, a term previously unknown to me. What if we could chop our search parameters in half? That would save time. Even better, let's keep chopping those halves in half until we get a really small sample to work with. That would really save unnecessary computer cycles.

But how do we decide which half of our collection to keep for the search, and which half to discard? This method of binary search only works, of course, when a collection is sorted, from A to Z, or 1 to 100. Imagine searching the phone book for Luigi's Pizza. You're not going to start at page 1 and scan each page. Crack open to a page somewhere near the middle, and start moving forward or back depending on where you land.

With some trial and error, I came to this iteration, which suited me well.

def binary_search(array, name)
  begin_search = 0
  end_search = array.length
  loop do
    middle = (end_search + begin_search) / 2
    if name == array[middle].first_name
      p "match found at #{middle}"
      return
    elsif name > array[middle].first_name
      begin_search = middle + 1
    else
      end_search = middle
    end
  end
end

The basic strategy is to test the target term against the item in the middle. Less than that? We focus on the first half. Greater than, the latter half. We do this by setting a start point and end point for the search, and calculate the middle. We winnow down the range by adjusting the start, end and middle points.

It's kind of amazing to see how few times the loop needs to run until you hit the mark. Search for a number between 1 and 50, and it's found within just a few runs through the loop. Imagine running a linear search for '20' and you know exactly how many times it'll run. 19 fails before success.

One nice thing about an implementation like this is that it has some universal application to a search based on numbers or letters.
